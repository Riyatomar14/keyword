### 2.1 PHYSICAL NIC - The Hardware Gateway

```
     Real World Network
            |
            | (Ethernet cable)
            |
            ▼
    ┌───────────────┐
    │  Physical NIC │  ← Real hardware chip
    │   (eth0)      │
    └───────┬───────┘
            │
            │ (writes to RAM via DMA)
            │
            ▼
    ┌───────────────┐
    │  DMA Buffer   │  ← Special memory area
    │   in RAM      │
    └───────┬───────┘
            │
            │ (fires IRQ - interrupt signal)
            │
            ▼
```

**What is happening here:**

**Physical NIC** = A real hardware card plugged into your server
- Has a MAC address (like `00:1A:2B:3C:4D:5E`)
- Connected to actual cables
- Receives electrical signals or light pulses

**What it does:**
1. Frame arrives on the wire
2. NIC writes it directly to RAM (DMA = Direct Memory Access, no CPU needed)
3. NIC sends an **interrupt** to the CPU: "Hey! New packet arrived!"

**Why this matters:**
- This is the ONLY real hardware in the entire chain
- Everything else is software pretending to be hardware
- All virtual NICs ultimately share THIS one physical card

---

### 2.2 LINUX KERNEL - Packet Enters Software

```
            IRQ fires
               |
               ▼
    ┌─────────────────────┐
    │   Interrupt Handler │
    │   (CPU wakes up)    │
    └──────────┬──────────┘
               │
               ▼
    ┌─────────────────────┐
    │   NIC Driver        │  ← Software that talks to hardware
    │   (e1000, ixgbe)    │
    └──────────┬──────────┘
               │
               │ Reads from DMA buffer
               │
               ▼
    ┌─────────────────────┐
    │   Creates sk_buff   │  ← The kernel's packet container
    │   (socket buffer)   │
    └──────────┬──────────┘
               │
               │
               ▼
    ┌─────────────────────┐
    │ netif_receive_skb() │  ← Entry to network stack
    │                     │
    └──────────┬──────────┘
               │
               ▼
      Packet is now in the kernel!
```

**What is happening here:**

**sk_buff** = The kernel's way of storing a network packet
- Contains the raw packet bytes
- Plus metadata: which interface it came from, timestamp, etc.

**Key point:**
From this moment on, the packet exists as a **software object** inside Linux kernel memory.

### 2.3 VIRTUAL NIC (veth pair) - The Magic Cable

```
┌──────────────────────────┐      ┌──────────────────────────┐
│   HOST NAMESPACE         │      │   CONTAINER NAMESPACE    │
│   (root network stack)   │      │   (isolated network)     │
│                          │      │                          │
│         veth0            │      │          veth1           │
│           │              │      │            │             │
│    (host side of         │      │     (container side)     │
│     virtual cable)       │      │                          │
│           │              │      │     Renamed to: eth0     │
│           │              │      │                          │
│  Attached to bridge/OVS  │      │     Has Pod IP           │
│           │              │      │     10.244.1.5/24        │
└───────────┼──────────────┘      └────────────┼─────────────┘
            │                                   │
            │    Virtual "cable" connection     │
            └───────────────┬───────────────────┘
                            │
                      (anything in veth0 
                       comes out veth1)
```

**What is a veth pair:**

Think of it as a **virtual Ethernet cable** with two ends:
- **veth0** stays on the host
- **veth1** goes into the container

**Key concept:**
When you send a packet into `veth0`, it **magically appears** at `veth1`.

**Network Namespace** = A completely isolated copy of the network stack.

```
┌─────────────────────────────────────────────────────────┐
│                     HOST MACHINE                        │
│                                                         │
│  ┌─────────────────┐         ┌─────────────────┐      │

│  │  (Container A)  │         │  (Container B)  │      │

│  │  eth0: 10.1.1.2 │         │  eth0: 10.1.1.3 │      │

│  │  iptables       │         │  iptables       │      │
│  └─────────────────┘         └─────────────────┘      │
│                                                         │
│  ┌─────────────────────────────────────────────┐       │
│  │        Root Namespace (Host)                │       │
│  │        eth0: physical NIC                   │       │
│  │        br0: bridge                          │       │
│  └─────────────────────────────────────────────┘       │
└─────────────────────────────────────────────────────────┘
```

### 2.4 BRIDGE / OVS - The Software Switch

```
                    Linux Bridge (br0)
                           │
           ┌───────────────┼───────────────┐
           │               │               │
           ▼               ▼               ▼
        veth0           veth2           veth4
           │               │               │
           │               │               │
      (to Pod A)      (to Pod B)      (to Pod C)
```

**What is a Bridge:**

A **Layer 2 switch** implemented in software.

**What it does:**
1. Receives frames from attached interfaces (veth ports, physical NICs)
2. Looks at destination MAC address
3. Forwards frame to the correct port

**It's like a traffic cop** directing packets to the right destination.

**Linux Bridge vs OVS:**

```
┌──────────────────┐              ┌──────────────────┐
│  Linux Bridge    │              │       OVS        │
├──────────────────┤              ├──────────────────┤
│                  │              │                  │
│ • Simple         │              │ • Programmable   │
│ • Built-in       │              │ • OpenFlow       │
│ • Good for basic │              │ • SDN-ready      │
│   setups         │              │ • High perf      │
│                  │              │   (with DPDK)    │
│ • Used by:       │              │ • Used by:       │
│   Flannel        │              │   OpenStack      │
│   Simple Docker  │              │   Large clouds   │
└──────────────────┘              └──────────────────┘
```

### 2.5 CNI - The Setup Worker

```
                    Kubernetes creates a Pod
                            │
                            ▼
                    kubelet calls CNI plugin
                            │
            ┌───────────────┼───────────────┐
            │               │               │
            ▼               ▼               ▼
        Flannel         Calico          Cilium
        
        Each CNI plugin's job:
        
        1. Create network namespace
        2. Create veth pair
        3. Move one end into namespace
        4. Attach other end to bridge
        5. Assign IP address (IPAM)
```

**What is CNI:**

**CNI = Container Network Interface**

It's a **standard specification** that says:

> "When you create a container, here's how you should set up its networking."

**Why CNI exists:**

**Kubernetes does NOT do networking itself.**

Instead, it says:
> "Hey CNI plugin, set up networking for this Pod. I don't care HOW you do it, just make it work."

## 3. WHY eBPF? WHY CILIUM?

### The Problem with Traditional Networking (iptables)

```
        With 10,000 Pods in a cluster:
        
┌─────────────────────────────────────────┐
│  iptables rule 1                        │
│  iptables rule 2                        │
│  iptables rule 3                        │
│  ...                                    │
│  iptables rule 9,997                    │
│  iptables rule 9,998                    │
│  iptables rule 9,999                    │  ← Must scan EVERY rule
│  iptables rule 10,000                   │     for EVERY packet!
└─────────────────────────────────────────┘

Every packet must scan ALL rules = O(n)
Slow! Gets worse with more Pods!
```

**Problem:**
- iptables uses a **linear list** of rules
- Every packet scans from top to bottom
- With thousands of Pods = thousands of rules
- Performance gets WORSE as cluster grows

### The eBPF Solution

```
        With 10,000 Pods in a cluster:
        
┌─────────────────────────────────────────┐
│                                         │
│         eBPF Hash Map                   │
│                                         │
│    Key: Pod IP    →  Value: Rules      │
│    10.244.1.5     →  identity=1234     │
│    10.244.1.6     →  identity=5678     │
│    ...                                  │
│                                         │
└─────────────────────────────────────────┘

Packet arrives → Hash lookup → O(1) constant time!
No matter if you have 10 Pods or 10,000 Pods!
```

**eBPF advantage:**
- Uses **hash tables** for lookups
- O(1) = constant time, doesn't matter how many Pods
- Dramatically faster at scale

### What is eBPF

```
        Traditional Linux Kernel
        
┌─────────────────────────────────────────┐
│         Kernel Code                     │
│    (written in C, compiled in)          │
│                                         │
│    To change = recompile kernel         │
│                 or                      │
│    Write kernel module (risky!)         │
└─────────────────────────────────────────┘
```

```
        With eBPF
        
┌─────────────────────────────────────────┐
│         Kernel Code                     │
│    (original kernel untouched)          │
│                                         │
│    ┌─────────────────────┐             │
│    │  eBPF Program 1     │  ← You write this!
│    │  (runs in kernel)   │  ← Safe, verified
│    └─────────────────────┘  ← Hot-loaded
│                                         │
│    ┌─────────────────────┐             │
│    │  eBPF Program 2     │             │
│    └─────────────────────┘             │
└─────────────────────────────────────────┘
```

**eBPF = Extended Berkeley Packet Filter**

A way to run custom programs **inside the Linux kernel** without:
- Recompiling the kernel
- Rebooting the system
- Risking kernel crashes

### Where eBPF Hooks Into the Kernel

```
Packet Journey with eBPF:

Physical NIC
    │
    ▼
┌─────────────────────┐
│  XDP Hook           │ ← eBPF can attach here (earliest!)
│  (eXpress Data Path)│    Can drop packets instantly
└──────────┬──────────┘
           │
           ▼
┌─────────────────────┐
│  TC Hook            │ ← eBPF can attach here
│  (Traffic Control)  │    Can modify, redirect packets
└──────────┬──────────┘
           │
           ▼
┌─────────────────────┐
│  Netfilter          │ ← eBPF can attach here
│  (iptables layer)   │
└──────────┬──────────┘
           │
           ▼
┌─────────────────────┐
│  Socket Layer       │ ← eBPF can attach here
│                     │    L7 inspection possible!
└─────────────────────┘
```

### Why Cilium Uses eBPF - 4 Key Reasons

**Problem 1: iptables is too slow**

```
Old way (kube-proxy with iptables):
Packet arrives → Check all rules (O(n))

Cilium way (eBPF):
Packet arrives → Hash lookup (O(1))
```

**Result:** 10-100x more Pods at the same speed.

**Problem 2: Pod IPs keep changing**

```
Traditional: Allow 10.244.1.5 to 10.244.2.8
Pod restarts → New IP → Rule BROKEN!

Cilium: Allow identity=1234 to identity=5678
Pod restarts → Same identity → Rule STILL WORKS!
```

**Problem 3: Can't see inside HTTP/gRPC**

```
Traditional: Only sees IP:Port

Cilium: Can see HTTP method, URL path, headers
Can enforce: "Allow only GET /api/public/*"
```

**Problem 4: No visibility**

```
Traditional: ¯\_(ツ)_/¯  Can't see what's happening

Cilium (Hubble): Real-time flow visibility!
See every packet, every decision
```

# PART 2: THE THREE FUNDAMENTAL TECHNIQUES

Every packet processing system uses one or more of these three core techniques:

1. **Packet Processing (PP)** — Examining and modifying packets
2. **Packet Bypass (PB)** — Avoiding kernel stack for performance
3. **Packet Filtering (PF)** — Allowing or denying packets

---

## 4. PACKET PROCESSING

### 4.1 Understanding CPU Privilege Levels

Before we dive into packet processing, you need to understand where code runs:

```
┌─────────────────────────────────────────────────────────┐
│                  CPU PRIVILEGE RINGS                    │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  Ring 0 (KERNEL MODE)                                  │
│  ┌──────────────────────────────────────┐              │
│  │ • Direct hardware access             │              │
│  │ • No memory protection               │              │
│  │ • Crash = system crash               │              │
│  │ • Examples: device drivers, kernel   │              │
│  └──────────────────────────────────────┘              │
│                                                         │
│  Ring 3 (USER MODE)                                    │
│  ┌──────────────────────────────────────┐              │
│  │ • No hardware access                 │              │
│  │ • Memory protected                   │              │
│  │ • Crash = only app crashes           │              │
│  │ • Examples: applications, services   │              │
│  └──────────────────────────────────────┘              │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### 4.2 Packet Processing in KERNEL MODE

```
Packet arrives at NIC → DMA to memory → Interrupt → Kernel takes over

Step-by-step in kernel:

1. HARDWARE INTERRUPT (hardirq context):
   - NIC raises interrupt line
   - CPU jumps to interrupt handler (NIC driver)
   - Driver acknowledges interrupt
   - Driver schedules NET_RX_SOFTIRQ
   - Returns from interrupt (~1-5µs)

2. SOFTIRQ CONTEXT (ksoftirqd):
   - net_rx_action() called
   - Driver's poll() function called (NAPI)
   - For each packet:
     a. Allocate sk_buff structure
     b. DMA packet data into sk_buff buffer
     c. Set sk_buff metadata
     d. Call netif_receive_skb(skb)

3. PACKET RECEIVE PROCESSING:
   - __netif_receive_skb_core()
   - Check for rx_handler
   - Deliver to protocol handler:
     - IP (0x0800) → ip_rcv()
     - ARP (0x0806) → arp_rcv()

4. IP LAYER PROCESSING (ip_rcv):
   - Validate IP header
   - Netfilter PREROUTING hook
   - Routing decision: ip_route_input()
     - Lookup in routing table (LC-trie)
     - Result: LOCAL or FORWARD

5a. IF LOCAL:
    - Netfilter LOCAL_IN hook
    - Deliver to TCP/UDP
    - Find socket (4-tuple lookup)
    - Copy to socket buffer
    - Wake up application

5b. IF FORWARD:
    - Netfilter FORWARD hook
    - Decrement TTL
    - ARP resolution
    - Netfilter POSTROUTING hook
    - Transmit
```

**Key Data Structure: sk_buff**

```c
struct sk_buff {
    struct sk_buff *next, *prev;       // Linked list
    struct net_device *dev;            // Interface
    unsigned char *head, *data, *tail, *end;  // Buffer pointers
    __u16 transport_header, network_header, mac_header;
    __be16 protocol;                   // Ethertype
    __u32 mark;                        // Netfilter mark
    struct dst_entry *dst;             // Routing decision
};
```

### 4.3 Kernel Mode Across Machine Types

#### PHYSICAL MACHINE

```
Flow: NIC → DMA → Kernel → Application

Hardware: Real NIC (Intel, Mellanox)
Driver: igb, ixgbe, mlx5_core
Process:
  1. NIC DMA to RAM (physical address)
  2. Interrupt to CPU
  3. Kernel allocates sk_buff
  4. Full kernel network stack
  5. Deliver to application

Performance: ~10-50µs latency, 5-10 Gbps per core
```

#### VIRTUAL MACHINE

```
Flow: Physical NIC → Hypervisor Kernel → vSwitch → VM Kernel → App

TWO KERNELS:
  - Host kernel (KVM/ESXi)
  - Guest kernel (inside VM)

Example: KVM with virtio-net

HOST SIDE:
  1. Physical NIC → host sk_buff
  2. Host kernel processes
  3. OVS/bridge forwards
  4. vhost-net writes to virtqueue

GUEST SIDE:
  5. VM receives interrupt
  6. virtio-net reads virtqueue
  7. Guest sk_buff allocated
  8. Guest kernel stack
  9. Deliver to app in VM

Performance: ~50-200µs (double kernel overhead)
```

#### CONTAINER

```
Flow: NIC → Kernel → veth → Container netns

SINGLE KERNEL, MULTIPLE NAMESPACES:

Process:
  1. Physical NIC → host sk_buff
  2. Host kernel routing
  3. Route to veth pair
  4. Packet crosses to container netns
  5. Same kernel, different namespace context
  6. Deliver to socket in container

Performance: ~10-30µs (veth ~5µs overhead)
Key: SAME KERNEL, just different namespaces
```

#### KUBERNETES POD

```
Flow: NIC → Kernel → CNI → veth → Pod netns

Process:
  1. Physical NIC → host kernel
  2. Routing to pod IP (CNI installed route)
  3. iptables NetworkPolicy check
  4. veth pair crossing
  5. Pod netns delivery
  6. Application recv()

Performance: ~20-100µs (iptables + veth)
```

### 4.4 Packet Processing in USER MODE

```
Application needs packets → syscall → kernel

RECEIVING:

1. Application calls recv():
   recv(sockfd, buffer, size, 0);  ← SYSCALL

2. SYSCALL MECHANISM:
   - CPU switches Ring 3 → Ring 0
   - Save user context
   - Jump to kernel handler
   - Syscall overhead: ~100-200ns

3. IN KERNEL:
   - Look up socket by fd
   - Check socket buffer
   - If data available: copy to user buffer
   - If no data: sleep until packet arrives
   - Return to userspace

4. Return to user:
   - Restore context
   - Application continues

ZERO-COPY OPTIMIZATIONS:

sendfile():
  - Kernel reads file directly to socket
  - No userspace copy

io_uring:
  - Shared ring buffers
  - Batches syscalls
  - ~1-2µs vs ~5-10µs traditional
```

### 4.5 User Mode Across Machine Types

#### PHYSICAL MACHINE
```
nginx web server example:

1. nginx socket(), bind(), listen()
2. Client connects (kernel TCP handshake)
3. nginx epoll_wait() blocks
4. Packet arrives → kernel → socket buffer
5. Kernel wakes nginx
6. nginx recv() → copy to nginx buffer
7. nginx processes HTTP
8. nginx sendfile() → kernel sends

Performance: ~50-100µs application latency
```

#### VIRTUAL MACHINE
```
Application in VM:

1. App in VM: socket(), bind(), listen()
2. Packet: Physical → Host → OVS → VM
3. Guest kernel → guest socket buffer
4. App recv() → guest kernel copies
5. App processes in VM userspace

Performance: ~100-300µs
Overhead: Guest syscall + host processing
```

#### CONTAINER
```
nginx in Docker:

1. Container netns created
2. nginx socket(), bind(80) in container netns
3. Packet: NIC → docker0 → veth → container
4. Host kernel delivers (in container netns context)
5. nginx recv() → syscall to same host kernel
6. nginx processes

Performance: ~50-150µs
Same kernel, different namespace
```

#### KUBERNETES POD
```
Flask app in pod:

1. Pod created, CNI assigns IP
2. Service created with ClusterIP
3. kube-proxy installs iptables DNAT rules
4. Client → ClusterIP → DNAT → Pod IP
5. Routing → veth → pod netns
6. Flask recv() → syscall
7. Flask processes

Performance: ~100-200µs (iptables + veth)
```

## 5. PACKET BYPASS

### 5.1 What is Bypass Mode?

Techniques that avoid the standard kernel network stack:
- **DPDK** - Userspace poll-mode drivers
- **XDP** - Kernel bypass at driver level
- **Hardware Offload** - NIC does processing

### 5.2 DPDK (Data Plane Development Kit)

```
KEY CONCEPT: Application directly accesses NIC, kernel is OUT

Architecture:
  Application → DPDK libraries → PMD → NIC registers → NIC
```

**DPDK Setup:**

```bash
# 1. Reserve hugepages (2MB or 1GB)
echo 1024 > /sys/kernel/mm/hugepages/hugepages-2048kB/nr_hugepages
# Why: Reduces TLB misses, better DMA

# 2. Unbind NIC from kernel
echo 0000:01:00.0 > /sys/bus/pci/drivers/ixgbe/unbind
# Why: Kernel can't interfere

# 3. Bind to VFIO/UIO
echo vfio-pci > /sys/bus/pci/devices/0000:01:00.0/driver_override
echo 0000:01:00.0 > /sys/bus/pci/drivers/vfio-pci/bind
# Why: Userspace PCI access

# 4. Initialize DPDK
rte_eal_init(argc, argv);
```

**DPDK Packet Flow:**

```
1. NIC receives → DMA to hugepage memory
2. NIC updates RX descriptor ring
3. DPDK app POLLS (while(1), CPU 100%):
   uint16_t nb_rx = rte_eth_rx_burst(port, queue, mbufs, BURST);
   - No interrupt
   - No kernel
   - No syscall
   
4. App gets rte_mbuf:
   struct rte_mbuf {
       void *buf_addr;
       rte_iova_t buf_iova;
       uint16_t pkt_len;
       ...
   };
   
5. Process packet (parse, modify)
6. Transmit:
   rte_eth_tx_burst(port, queue, &mbuf, 1);
```

**Performance:**
- Latency: 1-5µs
- Throughput: 10-24 Mpps per core
- CPU: 100% busy (acceptable for dedicated appliances)

### 5.3 DPDK Across Machine Types

#### PHYSICAL MACHINE
```
Dedicated server with DPDK VNF router:

- 2x 25G NICs bound to DPDK
- Cores isolated for DPDK
- Kernel on other cores

Flow:
1. Packet → NIC0 → hugepage
2. DPDK polls
3. Route lookup (hash table)
4. TX burst → NIC1

Performance: 40+ Mpps, <3µs latency
Use: NFV routers, firewalls
```

#### VIRTUAL MACHINE
```
VM with vhost-user:

Setup:
- Host runs OVS-DPDK
- VM uses vhost-user (not virtio-net)
- Shared hugepage memory

Flow:
1. Physical NIC → Host OVS-DPDK
2. OVS writes to vhost-user ring
3. VM DPDK polls vhost-user ring
4. VM processes
5. VM TX → vhost-user ring
6. OVS-DPDK → physical NIC

Performance: 8-15 Mpps, ~10-30µs
Benefit: No VM exits for data
```

#### CONTAINER
```
DPDK in container (CNF):

Setup:

- Privileged mode

Same as physical DPDK
```

#### KUBERNETES

### 5.4 XDP (eXpress Data Path)

**What is XDP:**
- eBPF programs at NIC driver receive path
- Runs BEFORE sk_buff allocation
- Minimal overhead

```
Normal: NIC → DMA → sk_buff alloc → processing
                     ↑ ~100ns overhead

XDP:    NIC → DMA → XDP program → decision
                     ↑ ~50ns total
```

**XDP Actions:**

```c
XDP_DROP:       Drop immediately (DDoS)
XDP_PASS:       Continue to kernel
XDP_TX:         Reflect back (simple switch)
XDP_REDIRECT:   Send to different interface
XDP_ABORTED:    Error (malformed packet)
```

**XDP Program Example:**

```c
#include <linux/bpf.h>
#include <linux/ip.h>

// Blacklist map
struct {
    __uint(type, BPF_MAP_TYPE_HASH);
    __uint(max_entries, 1000000);
    __type(key, __u32);    // Source IP
    __type(value, __u8);
} blacklist SEC(".maps");

SEC("xdp")
int xdp_drop_blacklist(struct xdp_md *ctx) {
    void *data = (void *)(long)ctx->data;
    void *data_end = (void *)(long)ctx->data_end;
    
    struct ethhdr *eth = data;
    if ((void *)(eth + 1) > data_end)
        return XDP_ABORTED;
    
    if (eth->h_proto != htons(ETH_P_IP))
        return XDP_PASS;
    
    struct iphdr *ip = (void *)(eth + 1);
    if ((void *)(ip + 1) > data_end)
        return XDP_ABORTED;
    
    // Check blacklist
    if (bpf_map_lookup_elem(&blacklist, &ip->saddr))
        return XDP_DROP;  // Drop at driver level
    
    return XDP_PASS;
}
```

```bash
# Compile and load
clang -O2 -target bpf -c xdp_drop.c -o xdp_drop.o
ip link set dev eth0 xdp obj xdp_drop.o sec xdp

# Add IP to blacklist
bpftool map update name blacklist key 0x01020304 value 0x00
```

**Performance:**
- DROP: ~50-100ns, 20-24 Mpps
- PASS: Same as normal kernel
- TX: ~100-200ns, 10-20 Mpps

### 5.5 XDP Across Machine Types

#### PHYSICAL MACHINE
```
XDP on physical NIC:

Flow:
1. Packet → NIC DMA
2. Driver calls XDP before netif_receive_skb
3. XDP program executes (JIT-compiled)
4. Based on action (DROP/PASS/TX/REDIRECT)
```

#### VIRTUAL MACHINE
```

   - Moderate performance

   - Better for dropping before VM

```

#### CONTAINER
```

Setup:
- Attach to host-side veth

Flow:
1. Packet to container → veth-abc
2. XDP runs before veth TX
3. DROP or PASS

Note: "Generic XDP" (software, not driver-level)
Performance: ~200-500ns (slower than native XDP)
```

#### KUBERNETES
```
Cilium with XDP:

Flow (NodePort service):
1. External packet → node IP:port
2. XDP on physical NIC
3. XDP checks: NodePort service?
4. If yes: XDP REDIRECT to pod veth
   - No iptables

```

### 5.6 Hardware Offload

**SmartNIC / Hardware OVS:**

```
Packet processing in NIC hardware (ASIC/FPGA)

Example: Mellanox ConnectX-5/6

Capability:
- Embedded switch (eSwitch)
- TC flower offload
- OVS rules in NIC hardware

5. Host CPU sees nothing!

Performance: Line rate (100 Gbps), <1µs
CPU: ~0% for forwarded traffic
```

## 6. PACKET FILTERING

### 6.1 iptables / Netfilter (Kernel Mode)

```
Netfilter = kernel framework
iptables = userspace tool

Architecture:
- 5 hooks: PREROUTING, INPUT, FORWARD, OUTPUT, POSTROUTING
- Tables: filter, nat, mangle, raw
- Chains: built-in or custom
- Rules: match + action

Rule evaluation: O(n) linear scan
```

**Example Firewall:**

```bash
iptables -A INPUT -p tcp --dport 22 -j ACCEPT
iptables -A INPUT -p tcp --dport 80 -j ACCEPT
iptables -A INPUT -m conntrack --ctstate ESTABLISHED,RELATED -j ACCEPT
iptables -A INPUT -j DROP
```

**Packet Processing:**

```
1. Packet arrives → PREROUTING hook
2. For each rule in chain:
   - Match protocol, port, state
   - If match: execute action
   - If no match: next rule
3. Routing decision
4. INPUT/FORWARD hook (same rule walk)

Performance:
- 10 rules: ~1µs
- 100 rules: ~5-10µs
- 1000 rules: ~50-100µs
- 10,000 rules: Minutes to load, unusable
```

### 6.2 Connection Tracking (conntrack)

```
Stateful firewall: remember connection state

How it works:
1. First packet (SYN):
   - Create conntrack entry
   - 5-tuple → state=NEW
   
2. Reply (SYN-ACK):
   - Update: NEW → ESTABLISHED
   
3. Subsequent packets:
   - Lookup: ESTABLISHED
   - Single rule accepts all!
   
4. FIN or timeout:
   - Remove entry

Performance:
- Hash lookup: ~500ns
- State update: ~200ns
- Total: ~1-2µs per packet

Table exhaustion:
- Default: 65,536 entries
- High churn: millions of connections
- Solution: Increase limit or use eBPF
```

### 6.3 iptables Across Machine Types

#### PHYSICAL MACHINE
```bash
# Web server firewall
iptables -P INPUT DROP
iptables -A INPUT -i lo -j ACCEPT
iptables -A INPUT -m conntrack --ctstate ESTABLISHED,RELATED -j ACCEPT
iptables -A INPUT -p tcp --dport 22 -s 192.168.1.0/24 -j ACCEPT
iptables -A INPUT -p tcp --dport 80 -j ACCEPT
iptables -A INPUT -p tcp --dport 443 -j ACCEPT
iptables -A INPUT -j DROP

# Performance: ~2-5µs overhead
```

#### VIRTUAL MACHINE
```
Two layers of filtering:

Host: iptables on tap device
Guest: iptables in VM

Example:
Host:  iptables -A FORWARD -o tap0 -p tcp --dport 3389 -j DROP
Guest: iptables -A INPUT -p tcp --dport 22 -j ACCEPT

Packet filtered TWICE
```

#### CONTAINER
```
Docker automatically creates rules:

docker run -p 8080:80 nginx

Creates:
iptables -t nat -A DOCKER -p tcp --dport 8080 \
  -j DNAT --to-destination 172.17.0.2:80
  
iptables -A DOCKER -d 172.17.0.2 -p tcp --dport 80 -j ACCEPT

Performance degradation:
- 10 containers: ~10µs
- 100 containers: ~50-100µs
```

#### KUBERNETES
```
NetworkPolicy via Calico:

apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: deny-all-allow-http
spec:
  podSelector: {app: web}
  ingress:

iptables -A cali-tw-web-pod -j DROP

Performance with 1000 pods:
- IPSet lookup: O(1), ~100ns
- Total: ~5-10µs
- Much better than 1000 individual rules
```

### 6.4 User-Mode Filtering

**Application-level filtering:**

#### nginx geo-blocking
```nginx

    default 0;
    91.0.0.0/8 1;    # Block Russia
    112.0.0.0/8 1;
}

    if ($block_country) {

```

Flow:
1. TCP connection established (kernel allows)
2. nginx accepts
3. nginx reads client IP
4. nginx checks geo database
5. If blocked: send 403

Performance: ~100µs overhead
Why use: Layer 7 awareness

#### Squid proxy ACLs
```

```

All in userspace, full protocol parsing

### 6.5 eBPF Filtering (Bypass Mode)

```
Advantages over iptables:
- O(1) map lookups vs O(n) chains
- Runs at driver (XDP) or TC
- Programmable custom logic

Performance:
- eBPF: ~100ns per packet
- iptables: ~5µs for 100 rules
- 50x faster
```

#### Cilium eBPF NetworkPolicy

```c
SEC("tc/ingress")
int policy_check(struct __sk_buff *skb) {
    __u32 src_identity = skb->cb[CB_SRC_ID];
    __u32 dst_identity = skb->cb[CB_DST_ID];
    
    struct policy_key key = {
        src_identity, dst_identity, dst_port, proto
    };
    
    struct policy_val *val = bpf_map_lookup_elem(&policy_map, &key);
    
    if (!val || val->action == DROP)
        return TC_ACT_SHOT;
    
    return TC_ACT_OK;
}
```

Performance: O(1) hash lookup, ~100ns

```
Linux Kernel    │  sk_buff, routing          │  Ring 0 (kernel)
Three Paths     │  PP / PB / PF              │  Varies
veth Pair       │  Virtual cable             │  Ring 0 (kernel)
Bridge/OVS      │  Software switch           │  Ring 0 (kernel)
CNI             │  Setup networking          │  Ring 3 (user)
Container/Pod   │  Application runs          │  Ring 3 (user)
```

### Three Fundamental Techniques

1. **Packet Processing (PP)**
   - Standard kernel path
   - Used: Everywhere by default
   - Performance: Good (~10-100µs)

2. **Packet Bypass (PB)**
   - DPDK, XDP, Hardware Offload
   - Used: High-performance scenarios
   - Performance: Excellent (~1-10µs)

3. **Packet Filtering (PF)**
   - iptables, eBPF
   - Used: Security, policies
   - Performance: iptables O(n), eBPF O(1)

### Machine Types

1. **Physical**: Single kernel, direct hardware
2. **VM**: Two kernels, virtualized hardware
3. **Container**: Single kernel, multiple namespaces
4. **K8s Pod**: Container + CNI + policies

### Why eBPF/Cilium

1. **Speed**: O(1) vs O(n)
2. **Identity**: Survives pod restarts
3. **L7**: Application-aware
4. **Visibility**: Hubble observability

## LEVEL 1: PHYSICAL NIC (Hardware Level)

### What Is This Level?

This is REAL HARDWARE - the actual network interface card physically plugged into your server.

```
┌─────────────────────────────────────────────────────────────┐
│                    PHYSICAL NIC CHIP                        │
│                                                             │
│  Ethernet Port ─────> NIC ASIC ─────> DMA Engine           │
│  (cable plugged)      (processes      (writes to           │
│                        packet)         RAM directly)        │
│                                                             │
│  Components:                                                │
│  • PHY: Converts electrical signals to bits                │
│  • MAC: Media Access Control, handles Ethernet frames      │
│  • DMA: Writes packet directly to RAM (no CPU!)            │
│  • Interrupt Controller: Signals CPU when packet arrives   │
└─────────────────────────────────────────────────────────────┘
```

### What Happens Here?

```
Step 1: Packet arrives on wire (electrical signals)
        ↓
Step 2: PHY chip converts signals to digital bits
        ↓
Step 3: MAC chip validates Ethernet frame (CRC check)
        ↓
Step 4: DMA engine writes packet to RAM buffer
        • No CPU involvement yet!
        • This is called "Direct Memory Access"
        • RAM location: Pre-allocated ring buffer
        ↓
Step 5: NIC raises hardware interrupt (IRQ)
        • Sends signal to CPU: "Hey! New packet!"
        • CPU stops what it's doing
        • CPU jumps to interrupt handler
```

### Technique Used: NONE (Pure Hardware)

**Why no technique?**
- This is physical hardware, not software
- No processing, filtering, or bypass decisions yet
- Just hardware doing hardware things

### Analogy

Think of it like mail delivery:
- **Physical NIC** = Your mailbox
- **DMA** = Mailman puts letter directly in your mailbox
- **Interrupt** = Mailbox flag goes up to alert you

## LEVEL 2: KERNEL - DRIVER LAYER

### What Is This Level?

This is the **device driver** - software that talks to the NIC hardware.

```
┌─────────────────────────────────────────────────────────────┐
│              LINUX KERNEL - DRIVER CODE                     │
│                                                             │
│  NIC Driver (e.g., ixgbe, i40e, mlx5_core)                 │
│                                                             │
│  Interrupt Handler:                                         │
│  1. CPU receives interrupt                                  │
│  2. Jump to driver's interrupt handler                      │
│  3. Driver reads NIC registers                              │
│  4. Driver schedules softirq (NET_RX_SOFTIRQ)               │
│                                                             │
│  SoftIRQ Handler:                                           │
│  5. Poll NIC's RX ring buffer                               │
│  6. Allocate sk_buff (kernel packet structure)              │
│  7. Copy packet from DMA buffer to sk_buff                  │
│  8. Fill sk_buff metadata                                   │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### What Happens Here?

```
HARDIRQ Context (very fast, ~1-5µs):
──────────────────────────────────────
1. CPU receives interrupt from NIC
   • CPU saves current work
   • CPU jumps to interrupt handler address
   
2. Driver interrupt handler runs:
   ixgbe_msix_clean_rings() {
       // Acknowledge interrupt to NIC
       // Schedule NET_RX_SOFTIRQ
       // Return immediately
   }

Why so fast? 
- Interrupts block other work
- Must finish ASAP
- Real work done in softirq


SOFTIRQ Context (slower, ~10-50µs):
────────────────────────────────────
3. Softirq runs later (or immediately if CPU idle):
   
   net_rx_action() {
       // Call driver's poll function
       ixgbe_poll()
   }
   
4. Driver poll function:
   
   ixgbe_poll() {
       // Read RX descriptor ring
       // For each packet in ring:
       
       while (packets_to_process) {
           // Allocate sk_buff
           skb = netdev_alloc_skb()
           
           // Copy packet data from DMA buffer
           memcpy(skb->data, dma_buffer, length)
           
           // Fill metadata
           skb->dev = netdev            // Which interface
           skb->protocol = eth_type     // 0x0800 = IPv4
           skb->len = packet_length
           skb->mac_header = 0
           skb->network_header = 14     // After Ethernet
           
           // Hand to network stack
           netif_receive_skb(skb)
       }
   }
```

### Technique Used: PACKET PROCESSING (PP)

**Why Packet Processing?**

Because we must:
1. **Read** packet from hardware buffer
2. **Convert** hardware data to software structure (sk_buff)
3. **Parse** basic headers (Ethernet type)
4. **Prepare** for network stack

This is **processing** - examining and preparing the packet.

### The sk_buff Structure

```c
struct sk_buff {
    // Linked list pointers
    struct sk_buff *next, *prev;
    
    // Interface packet arrived on
    struct net_device *dev;
    
    // Buffer pointers
    unsigned char *head;    // Start of buffer
    unsigned char *data;    // Start of actual data
    unsigned char *tail;    // End of data
    unsigned char *end;     // End of buffer
    
    // Header offsets
    u16 mac_header;        // Offset to Ethernet header
    u16 network_header;    // Offset to IP header
    u16 transport_header;  // Offset to TCP/UDP header
    
    // Protocol info
    __be16 protocol;       // Ethertype (0x0800 = IPv4)
    
    // Routing decision (filled later)
    struct dst_entry *dst;
    
    // Netfilter mark (for iptables)
    __u32 mark;
    
    // Length
    unsigned int len;
};
```

### Why This Matters

This is where the packet **becomes software**:
- Before: Just bytes in RAM
- After: Structured object kernel can work with

## LEVEL 3: KERNEL - NETWORK STACK ENTRY

### What Is This Level?

This is the **fork in the road** - packet can take three different paths!

```
┌─────────────────────────────────────────────────────────────┐
│         NETWORK STACK ENTRY POINT: THREE PATHS              │
│                                                             │
│              netif_receive_skb(skb)                         │
│                      │                                      │
│                      │                                      │
│         ┌────────────┼────────────┐                         │
│         │            │            │                         │
│         ▼            ▼            ▼                         │
│    ┌────────┐  ┌────────┐  ┌────────┐                      │
│    │  XDP   │  │Standard│  │ DPDK   │                      │
│    │(Bypass)│  │  Path  │  │(Bypass)│                      │
│    │        │  │  (PP)  │  │        │                      │
│    └────────┘  └────────┘  └────────┘                      │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### PATH 1: XDP (eXpress Data Path) - BYPASS

```
Location: Right after driver, BEFORE sk_buff allocation

┌─────────────────────────────────────┐
│  Driver receives packet             │
│         ↓                            │
│  eBPF XDP program runs              │
│  (attached to driver)                │
│         ↓                            │
│  Program decides:                    │
│  • XDP_DROP    → Drop immediately   │
│  • XDP_PASS    → Continue to kernel │
│  • XDP_TX      → Send back out      │
│  • XDP_REDIRECT → Send elsewhere    │
└─────────────────────────────────────┘
```

**When is XDP used?**

```
✓ DDoS protection (drop bad IPs instantly)
✓ Load balancing (redirect to backend)
✓ Fast filtering (drop before kernel overhead)
✓ Cilium uses this for fast service routing

Example:
  If packet from blacklisted IP:
    → XDP_DROP
    → No sk_buff allocated
    → No kernel processing
    → Dropped in ~50 nanoseconds!
```

**Technique: PACKET BYPASS (PB)**

Why bypass?
- Skips entire kernel network stack
- Drops/redirects before expensive processing
- 10-20x faster than normal path

### PATH 2: Standard Linux Network Stack - PROCESSING

```
Location: Normal kernel path

┌─────────────────────────────────────┐
│  netif_receive_skb(skb)             │
│         ↓                            │
│  __netif_receive_skb_core()         │
│         ↓                            │
│  Check rx_handler                    │
│  (Bridge, OVS, bonding)              │
│         ↓                            │
│  Deliver to protocol:                │
│  • ETH_P_IP (0x0800) → ip_rcv()     │
│  • ETH_P_ARP → arp_rcv()            │
│  • ETH_P_8021Q → VLAN handler       │
└─────────────────────────────────────┘
```

**When is Standard Path used?**

```
✓ Normal networking (99% of traffic)
✓ When no XDP program attached
✓ When packet passes XDP (XDP_PASS)
✓ Regular containers, VMs, applications
```

**Technique: PACKET PROCESSING (PP)**

Why processing?
- Must examine headers
- Must make routing decisions
- Must apply firewall rules
- Standard kernel features needed

### PATH 3: DPDK - COMPLETE BYPASS

```
Location: Driver doesn't even call netif_receive_skb!

┌─────────────────────────────────────┐
│  Application polls NIC directly     │
│         ↓                            │
│  No kernel involvement              │
│         ↓                            │
│  Application reads ring buffer      │
│         ↓                            │
│  Application processes packet       │
│         ↓                            │
│  Application transmits               │
└─────────────────────────────────────┘
```

**When is DPDK used?**

```
✓ NFV (Network Function Virtualization)
✓ High-frequency trading
✓ 5G core network
✓ When need 10M+ packets per second
```

**Technique: PACKET BYPASS (PB)**

Why bypass?
- Kernel is too slow for extreme performance
- Application takes complete control
- 100x faster than kernel for simple forwarding

### Which Path Does Our Packet Take?

```
For normal Kubernetes networking:
→ Standard Path (PACKET PROCESSING)

Exception: If Cilium with XDP:
→ XDP first (check if fast path possible)
→ If fast path: XDP_REDIRECT
→ If not: XDP_PASS → Standard Path
```

## LEVEL 4: KERNEL - IP LAYER

### What Is This Level?

This is where the kernel **understands the packet is IP** and makes routing decisions.

```
┌─────────────────────────────────────────────────────────────┐
│                    IP LAYER (ip_rcv)                        │
│                                                             │
│  1. Validate IP header                                      │
│     • Check version (IPv4 or IPv6)                          │
│     • Verify checksum                                       │
│     • Check TTL > 0                                         │
│                                                             │
│  2. Netfilter PREROUTING hook                               │
│     • iptables rules run here                               │
│     • Can modify packet (DNAT)                              │
│                                                             │
│  3. Routing decision                                        │
│     • Look up destination IP in routing table               │
│     • Decision: LOCAL or FORWARD?                           │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### What Happens Here?

```
Step 1: ip_rcv() function called
────────────────────────────────

ip_rcv(struct sk_buff *skb) {
    struct iphdr *iph = ip_hdr(skb);
    
    // Validate IP header
    if (iph->version != 4)
        return DROP;
        
    if (ip_fast_csum(iph) != 0)  // Bad checksum
        return DROP;
        
    if (iph->ttl == 0)  // TTL expired
        return DROP;
    
    // Continue to Netfilter
    return NF_HOOK(NFPROTO_IPV4, NF_INET_PRE_ROUTING, ...);
}


Step 2: Netfilter PREROUTING
─────────────────────────────

NF_HOOK() calls all registered hooks:
• iptables rules in PREROUTING chain
• Connection tracking (conntrack)
• NAT rules (DNAT happens here)

Example:
  iptables -t nat -A PREROUTING -p tcp --dport 8080 \
    -j DNAT --to-destination 10.0.0.5:80
    
  If packet to port 8080:
    → Change destination to 10.0.0.5:80
    → Modify skb->dst
    

Step 3: Routing Decision
─────────────────────────

ip_route_input(skb) {
    // Look up destination IP in FIB (Forwarding Information Base)
    dst = fib_lookup(skb->dst_ip);
    
    if (dst == LOCAL) {
        // Packet is for this machine
        skb->dst = local_route;
        return LOCAL;
    } else {
        // Packet needs forwarding
        skb->dst = forward_route;
        return FORWARD;
    }
}

Example routing table:

Destination      Gateway         Interface    Type
─────────────────────────────────────────────────────
10.244.0.0/16    0.0.0.0         cni0         LOCAL
10.244.1.5/32    0.0.0.0         veth-abc     LOCAL ← Our pod!
0.0.0.0/0        192.168.1.1     eth0         FORWARD
```

### Technique Used: PACKET PROCESSING (PP)

**Why Processing?**

Must perform:
1. **Parse** IP header (read destination, TTL, etc.)
2. **Validate** checksum, version, TTL
3. **Look up** routing table (LC-trie lookup, ~O(log n))
4. **Decide** local delivery vs forwarding

This is all **processing** - examining packet to make decisions.

### Real Example: Pod Traffic

```
Scenario: Packet to pod IP 10.244.1.5

Step 1: ip_rcv validates packet ✓

Step 2: Netfilter PREROUTING
  • No DNAT rules match
  • conntrack: Creates new connection entry
  • Packet continues

Step 3: Routing lookup
  • Destination: 10.244.1.5
  • Lookup routing table:
    → Match: 10.244.1.5/32 dev veth-abc
  • Decision: LOCAL (deliver to this machine)
  • Set skb->dst = route to veth-abc
```

## LEVEL 5: KERNEL - NETFILTER/IPTABLES

### What Is This Level?

This is the **firewall** - where security policies are enforced.

```
┌─────────────────────────────────────────────────────────────┐
│                  NETFILTER HOOKS                            │
│                                                             │
│   PREROUTING ──→ ROUTING ──→ FORWARD ──→ POSTROUTING       │
│                  DECISION        ↓                          │
│                                  ↓                          │
│                               INPUT                         │
│                                  ↓                          │
│                              LOCAL                          │
│                             PROCESS                         │
│                                  ↓                          │
│                               OUTPUT                        │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### What Happens Here?

```
Our packet (to pod 10.244.1.5) hits FORWARD hook:

NF_HOOK(NFPROTO_IPV4, NF_INET_FORWARD, skb) {
    // Run all iptables rules in FORWARD chain
    
    For each rule in iptables:
        1. Match packet against rule criteria
        2. If match: Execute action
        3. If no match: Continue to next rule
}

Example iptables rules (Calico NetworkPolicy):

# Jump to Calico chains
iptables -A FORWARD -j cali-FORWARD

# Calico forward chain
iptables -A cali-FORWARD -o veth-abc -j cali-to-pod-abc

# Per-pod chain (NetworkPolicy enforcement)
iptables -A cali-to-pod-abc \
  -m set --match-set cali-allowed-sources src \
  -p tcp --dport 80 \
  -j ACCEPT

iptables -A cali-to-pod-abc -j DROP


Packet Processing:
──────────────────

Packet from IP 192.168.1.5 to pod 10.244.1.5:80

Rule 1: -j cali-FORWARD
  → Jump to cali-FORWARD chain ✓

Rule 2: -o veth-abc -j cali-to-pod-abc
  → Output interface is veth-abc? Yes ✓
  → Jump to cali-to-pod-abc ✓

Rule 3: -m set --match-set cali-allowed-sources src ...
  → Is source IP in allowed set?
  → Check IPSet: ipset test cali-allowed-sources 192.168.1.5
  → If yes: ACCEPT ✓
  → If no: Continue

Rule 4: -j DROP
  → If we reach here: DROP packet
```

### Technique Used: PACKET FILTERING (PF)

**Why Filtering?**

This level decides:
1. **Allow** or **Drop** packet
2. **Modify** packet (SNAT, DNAT)
3. **Log** packet
4. **Mark** packet for later processing

This is pure **filtering** - making security decisions.

### Performance Concern

```
Problem with iptables:
  • Linear rule scanning: O(n)
  • 1000 pods = 1000s of rules
  • Each packet scans all rules
  • Slow!

Example timing:
  10 rules: ~1µs
  100 rules: ~10µs
  1000 rules: ~100µs
  10000 rules: ~1000µs (1ms per packet!)

Solution: Use eBPF (Cilium)
  • Hash table lookups: O(1)
  • Always ~100ns regardless of pod count
  • 100x faster!
```

### Alternative: eBPF Filtering

```
If using Cilium instead of iptables:

┌─────────────────────────────────────┐
│  TC (Traffic Control) hook          │
│         ↓                            │
│  eBPF program runs                   │
│         ↓                            │
│  Hash map lookup:                    │
│    key = (src_identity, dst_port)    │
│    value = allow/deny                │
│         ↓                            │
│  Decision in ~100ns                  │
└─────────────────────────────────────┘

eBPF program:

SEC("tc/egress")
int policy_check(struct __sk_buff *skb) {
    __u32 src_id = skb->cb[CB_SRC_IDENTITY];
    __u32 dst_port = get_port(skb);
    
    struct policy_key key = {src_id, dst_port};
    
    struct policy_val *val = 
        bpf_map_lookup_elem(&policy_map, &key);
    
    if (val && val->action == ALLOW)
        return TC_ACT_OK;
    
    return TC_ACT_SHOT;  // Drop
}

Performance:
  • Hash lookup: O(1)
  • ~100 nanoseconds
  • Same speed for 10 pods or 10,000 pods!
```

### Why This Level Exists

```
Security Requirements:
  ✓ Kubernetes NetworkPolicy
  ✓ Firewall rules
  ✓ Service mesh policies
  ✓ Compliance requirements

Without this level:
  ✗ No pod-to-pod isolation
  ✗ No ingress/egress control
  ✗ Security nightmare!
```

## LEVEL 6: SOFTWARE BRIDGE/OVS

### What Is This Level?

This is a **virtual switch** - like a physical network switch, but in software.

```
┌─────────────────────────────────────────────────────────────┐
│                  LINUX BRIDGE (br0)                         │
│                                                             │
│              ┌─────────────────┐                            │
│              │   MAC Table     │                            │
│              ├─────────────────┤                            │
│              │ MAC  │  Port    │                            │
│              ├─────────────────┤                            │
│              │ aa:bb │ veth1   │                            │
│              │ cc:dd │ veth2   │                            │
│              │ ee:ff │ veth3   │                            │
│              └─────────────────┘                            │
│                       │                                     │
│        ┌──────────────┼──────────────┐                      │
│        │              │              │                      │
│        ▼              ▼              ▼                      │
│     veth1          veth2          veth3                     │
│        │              │              │                      │
│        └──────────────┴──────────────┘                      │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### What Happens Here?

```
Step 1: Packet arrives at bridge
────────────────────────────────

Bridge receives packet on eth0 (physical interface)
Destination MAC: aa:bb:cc:dd:ee:ff
Destination IP: 10.244.1.5 (our pod)


Step 2: MAC Learning
────────────────────

Bridge maintains MAC address table:

┌──────────────────────────────────────┐
│  MAC Address    │  Port    │  Age   │
├──────────────────────────────────────┤
│  aa:bb:...:f1  │  veth1   │   2s   │ ← Pod 1
```

### Container Routing Table Explained

```
Why is the default route 169.254.1.1?

This is a "magic" IP that doesn't actually exist!

──────────────

1. Container wants to send packet outside

2. Routing lookup:
   No specific route for 8.8.8.8

3. ARP for 169.254.1.1:
   Container: "What's MAC of 169.254.1.1?"
   
4. Host responds with PROXY ARP:
   Host: "That's me! Here's my MAC"
   
5. Container sends packet to host's MAC
   → Goes through veth
   → Host receives and routes it
   
This is called "Point-to-Point" routing
No actual gateway at 169.254.1.1!
```

### What Happens Here?

```
Application State: Blocked waiting for connection

nginx code:
───────────

int main() {
    int sockfd = socket(AF_INET, SOCK_STREAM, 0);
    
    struct sockaddr_in addr = {
        .sin_family = AF_INET,
        .sin_port = htons(80),
        .sin_addr.s_addr = inet_addr("10.244.1.5")
    };
    
    bind(sockfd, &addr, sizeof(addr));
    listen(sockfd, 128);
    
    // Blocked here, waiting for connection
    int conn = accept(sockfd, NULL, NULL);
    
    // Still blocked, waiting for data
    char buffer[1024];
    int n = recv(conn, buffer, sizeof(buffer), 0);
    
    // Now we have data!
    // Process HTTP request
    parse_http_request(buffer, n);
}


What happens when packet arrives:
──────────────────────────────────

Step 1: Packet in socket buffer (kernel space)

struct sock *sk = nginx's listening socket;
sk->sk_receive_queue contains our packet


Step 2: Kernel wakes nginx

nginx process state: TASK_INTERRUPTIBLE (sleeping)
→ Kernel changes to: TASK_RUNNING
→ Scheduler adds nginx to run queue
→ CPU eventually runs nginx


Step 3: recv() syscall returns

recv() was blocked in kernel
Now it returns to userspace with data:

SYSCALL recv():
  1. Copy data from sk->sk_receive_queue
  2. Copy to user buffer (nginx's buffer)
  3. Return number of bytes copied
  
nginx's buffer now contains packet data!


Step 4: Application processes data

nginx parses HTTP:
  GET /index.html HTTP/1.1
  Host: example.com
  ...
  
nginx opens file /var/www/html/index.html
nginx sends response:
  HTTP/1.1 200 OK
  Content-Length: 1234
  
  <html>...</html>
```

### Technique Used: PACKET PROCESSING (PP)

**Why Processing?**

Must perform:
1. **Syscall** to kernel (recv)
2. **Copy** data from kernel to userspace
3. **Parse** application protocol (HTTP, gRPC, etc.)
4. **Process** business logic

This is **processing** - examining and handling data.

### Syscall Overhead

```
What happens in recv() syscall:

User Space (Ring 3):
  recv(sockfd, buffer, 1024, 0);
  ↓
  syscall instruction
  ↓
Kernel Space (Ring 0):
  ↓ (~100-200ns to switch rings)
  sys_recv() {
      // Find socket by fd
      sock = get_socket(sockfd);
      
      // Check if data available
      if (socket_has_data(sock)) {
          // Copy from kernel to user
          copy_to_user(buffer, sock->data, length);
          return length;
      } else {
          // No data: sleep
          sleep_on_socket(sock);
      }
  }
  ↓
  sysret instruction
  ↓
User Space:
  recv() returns
  
Total overhead:
  • Ring switch: ~200ns
  • Kernel work: ~500-1000ns
  • Memory copy: ~1-3µs (for small packets)
  • Total: ~2-5µs for syscall
```

### Zero-Copy Optimizations

```
Problem: Copying data is expensive!

Traditional path:
  NIC → DMA buffer → sk_buff → socket buffer → user buffer
  ↑                                             ↑
  Kernel memory                                 User memory

Zero-copy techniques:

1. sendfile():
   File → Kernel → NIC (no user copy)
   Used by: nginx serving static files
   
2. splice():
   Kernel pipe → Socket (no user copy)
   
3. io_uring:
   Shared memory between kernel and user
   No copy needed!
   
   ┌─────────────────────────────────┐
   │   Shared Ring Buffer            │
   │                                 │
   │   ┌───┬───┬───┬───┬───┐        │
   │   │ 1 │ 2 │ 3 │ 4 │ 5 │        │
   │   └───┴───┴───┴───┴───┘        │
   │    ↑               ↑            │
   │    │               │            │
   │  Kernel          User           │
   │  writes          reads          │
   └─────────────────────────────────┘
```

### Why This Level Exists

```
Separation of Concerns:
  
Kernel:
  ✓ Hardware management
  ✓ Memory protection
  ✓ Process isolation
  ✓ Network stack
  
Application:
  ✓ Business logic
  ✓ User interaction
  ✓ Data processing
  ✓ No hardware access (safer)
  
Benefits:
  • Security (app can't crash kernel)
  • Stability (bug in app doesn't break OS)
  • Portability (same app on different kernels)
```

### EXAMPLE 1: Calico NetworkPolicy (iptables - PACKET FILTERING)

```yaml
# Kubernetes NetworkPolicy
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-frontend-to-backend
spec:
  podSelector:
    matchLabels:
      app: backend
  ingress:
  - from:
    - podSelector:
        matchLabels:
          app: frontend
    ports:
    - protocol: TCP
      port: 8080
```

**Calico translates this to iptables:**

```bash
# Create IPSet for frontend pods
ipset create cali-frontend hash:ip
ipset add cali-frontend 10.244.1.5  # Frontend pod IP
ipset add cali-frontend 10.244.1.6

# Create iptables chain for backend pod
iptables -N cali-tw-backend-pod

# Rule 1: Allow from frontend pods on port 8080
iptables -A cali-tw-backend-pod \
  -p tcp --dport 8080 \
  -m set --match-set cali-frontend src \
  -j ACCEPT

# Rule 2: Drop everything else
iptables -A cali-tw-backend-pod -j DROP

# Jump to this chain for backend pod's veth
iptables -A FORWARD -o veth-backend -j cali-tw-backend-pod
```

**How it works:**

```
Packet from frontend (10.244.1.5) → backend (10.244.2.8):80

1. Kernel routing: Send to veth-backend
   ↓
2. Netfilter FORWARD hook triggers
   ↓
3. iptables chain: cali-tw-backend-pod
   ↓
4. Check Rule 1:
   • Source IP: 10.244.1.5
   • Is 10.244.1.5 in cali-frontend IPSet? YES ✓
   • Destination port: 80
   • Port 80 allowed? NO (only 8080 allowed) ✗
   ↓
5. Check Rule 2:
   • Match all: YES ✓
   • Action: DROP
   ↓
6. Packet DROPPED!

Result: Blocked by NetworkPolicy


Packet from frontend (10.244.1.5) → backend (10.244.2.8):8080

1-4. Same as above
   ↓
4. Check Rule 1:
   • Source IP in IPSet? YES ✓
   • Destination port 8080? YES ✓
   • Action: ACCEPT
   ↓
5. Packet ALLOWED!
   ↓
6. Continue to veth-backend
   ↓
7. Deliver to pod
```

**This is PACKET FILTERING (PF):**
- Examining packet
- Matching against policy rules
- Decision: ALLOW or DROP
- Security enforcement

### EXAMPLE 2: Cilium NetworkPolicy (eBPF - PACKET FILTERING)

**Same NetworkPolicy, but Cilium uses eBPF instead of iptables:**

```c
// eBPF program attached to veth-backend (TC ingress hook)

struct policy_key {
    __u32 src_identity;   // Identity of source pod
    __u16 dst_port;       // Destination port
    __u8  protocol;       // TCP/UDP
};

struct policy_val {
    __u8 action;  // ALLOW or DROP
};

// BPF map: policy rules
struct {
    __uint(type, BPF_MAP_TYPE_HASH);
    __type(key, struct policy_key);
    __type(value, struct policy_val);
} policy_map SEC(".maps");

SEC("tc/ingress")
int enforce_policy(struct __sk_buff *skb) {
    // Extract source identity (set by Cilium agent)
    __u32 src_identity = skb->cb[CB_SRC_IDENTITY];
    
    // Parse packet to get port
    struct iphdr *ip = /* parse IP header */;
    struct tcphdr *tcp = /* parse TCP header */;
    __u16 dst_port = tcp->dest;
    
    // Lookup in policy map
    struct policy_key key = {
        .src_identity = src_identity,
        .dst_port = dst_port,
        .protocol = IPPROTO_TCP
    };
    
    struct policy_val *val = bpf_map_lookup_elem(&policy_map, &key);
    
    if (!val) {
        // No policy match: default DROP
        return TC_ACT_SHOT;
    }
    
    if (val->action == ALLOW) {
        return TC_ACT_OK;  // Allow packet
    }
    
    return TC_ACT_SHOT;  // Drop packet
}
```

**Policy map contents (loaded by Cilium agent):**

```
Key: {src_identity=1234, dst_port=8080, proto=TCP}
Value: {action=ALLOW}

// 1234 = Identity for label app=frontend
// So: Traffic from app=frontend to port 8080: ALLOW
```

**How it works:**

```
Packet from frontend → backend:8080

1. Packet arrives at veth-backend
   ↓
2. TC (Traffic Control) ingress hook triggers
   ↓
3. eBPF program runs
   ↓
4. Extract src_identity from packet metadata
   • Cilium agent already labeled this packet with identity=1234
   ↓
5. Parse packet: dst_port = 8080
   ↓
6. Create lookup key: {1234, 8080, TCP}
   ↓
7. Hash map lookup: O(1) - instant!
   • Found: {action=ALLOW}
   ↓
8. Return TC_ACT_OK
   ↓
9. Packet continues to pod

Performance: ~100 nanoseconds
```

**This is ALSO PACKET FILTERING (PF):**
- Just using eBPF instead of iptables
- Same goal: enforce security policy
- Much faster: O(1) vs O(n)

**Example: Cilium Service Load Balancing with XDP:**

```c
// XDP program for ClusterIP service load balancing

SEC("xdp")
int service_loadbalancer(struct xdp_md *ctx) {
    void *data = (void *)(long)ctx->data;
    void *data_end = (void *)(long)ctx->data_end;
    
    struct ethhdr *eth = data;
    struct iphdr *ip = (void *)(eth + 1);
    struct tcphdr *tcp = (void *)(ip + 1);
    
    // Check bounds
    if ((void *)(tcp + 1) > data_end)
        return XDP_PASS;
    
    // Is this a ClusterIP service request?
    __be32 dest_ip = ip->daddr;
    __be16 dest_port = tcp->dest;
    
    struct service_key key = {dest_ip, dest_port};
    struct service_val *svc = bpf_map_lookup_elem(&services, &key);
    
    if (!svc) {
        // Not a service: normal processing
        return XDP_PASS;
    }
    
    // Service found! Pick backend pod (load balancing)
    __u32 backend_idx = bpf_get_prandom_u32() % svc->num_backends;
    struct backend *backend = &svc->backends[backend_idx];
    
    // Rewrite destination (like DNAT, but in XDP!)
    ip->daddr = backend->ip;
    tcp->dest = backend->port;
    
    // Recompute checksums
    ip->check = recalc_csum_diff(old_ip, new_ip, ip->check);
    tcp->check = recalc_csum_diff(old_tcp, new_tcp, tcp->check);
    
    // REDIRECT to backend pod's veth
    return bpf_redirect_map(&devmap, backend->ifindex, 0);
}
```

**How it works:**

```
Scenario: ClusterIP service 10.96.0.100:80 → 3 backend pods

External packet arrives: dest=10.96.0.100:80

1. NIC DMA to memory
   ↓
2. XDP program runs (BEFORE sk_buff allocation!)
   ↓
3. Check: Is 10.96.0.100:80 a service?
   • Lookup in services map
   • Found: 3 backends
   ↓
4. Pick random backend (load balance)
   • backend[1] = 10.244.1.5:8080
   ↓
5. Rewrite packet:
   • dest_ip: 10.96.0.100 → 10.244.1.5
   • dest_port: 80 → 8080
   ↓
6. XDP_REDIRECT to veth-pod1
   ↓
7. Packet appears at pod's veth

BYPASSED:
  ✗ sk_buff allocation
  ✗ Kernel network stack
  ✗ Routing table lookup
  ✗ Netfilter/iptables
  ✗ kube-proxy

Result: 10-50x faster than iptables!
```

**This is PACKET BYPASS (PB):**
- Avoiding kernel network stack
- Direct packet manipulation
- Extreme performance

   - By NetworkPolicy implementations
   - Location: Between routing and bridge

2. **Packet Bypass (PB)** - YES, it's used!
   - By Cilium XDP (early bypass)
   - By DPDK/OVS-DPDK (complete bypass)
   - By SmartNICs (hardware bypass)
   - Multiple locations possible

3. **Packet Processing (PP)** - YES, most common!
   - Default path
   - Most levels use this
   - Normal kernel networking

**The key insight:**
- **PP is the base** - everything starts here
- **PF is added** - for security (Calico, Cilium)
- **PB is optional** - for extreme performance (XDP, DPDK, SmartNIC)

All three can coexist in the same system!
